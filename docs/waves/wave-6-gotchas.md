# Wave-6 Gotchas

> Append-only log of surprises encountered while shipping wave-6 tasks.
> Format: one bullet per gotcha, with the date, task id, and one-paragraph "what broke + the fix".

- **2026-06-20 · wave-6/02-graders** — `evals/` lives at the repo root, NOT under `src/`, and has no `__init__.py`. With `PYTHONPATH=src` (the orchestrator's acceptance command), `import evals.graders` fails. Fix: prepend the repo root to `sys.path` at module-top in `tests/unit/test_graders.py`. `evals` becomes a PEP 420 namespace package, and `evals/graders/__init__.py` (which the W6-02 task owns) makes `evals.graders` resolvable. **Implication for downstream tasks**: any future task that imports from `evals.graders.*` (e.g., the W6-04 harness, the W6-05 report generator) MUST either (a) also prepend the repo root to `sys.path` at module-top, or (b) add `pythonpath = ["src", "."]` to the `pyproject.toml` `[tool.pytest.ini_options]` block. Recommend (b) in a tiny `tools/` follow-up so we don't repeat this in every test file. Also affects how `scripts/run_evals.py` (W6-04) must launch — `python -m scripts.run_evals` won't find `evals` unless the CWD is the repo root or `PYTHONPATH=.` is set.
- **2026-06-20 · wave-6/02-graders** — Pre-existing `evals/graders/{code_based,llm_judge}.py` and `human_review_template.md` were `NotImplementedError` placeholder stubs from OS-setup. They are in the W6-02 Writes set per the contract file map, so rewriting them is in-scope (not a collision). Future workers: if you open a graders file and see "STATUS: SCAFFOLD / INTERFACE ONLY" or `raise NotImplementedError(...)`, check your task's Writes set — if the path is listed, you're meant to replace it.
- **2026-06-20 · wave-6/04-harness** — The orchestrator populates `state.candidate` but leaves `state.final` None. The graders inspect `state.final`. The harness promotes `candidate → final` internally (`_ensure_final_populated`) rather than touching the orchestrator — keeping the W6-04 Writes set disjoint from the orchestrator's owner (a different wave). Future graders/reporters can do the same.
- **2026-06-20 · wave-6/04-harness** — `mock.py` was expanded to 55+ canned responses by an earlier wave, some containing digits (e.g. "45% equity, 35% debt"). The W6-03 `NaiveRAGBaseline.answer()` builds a `Recommendation(analysis=result.text, citations=[])` which now fails the FM-11 invariant (numeric analysis needs citations). NOT in this task's Writes set, so out of scope to fix; flagging for whoever owns the baselines. The harness itself handles this defensively (`_grade_trial` returns `score=0.0, passed=False` with an "error" breakdown instead of raising).
- **2026-06-20 · wave-6/04-harness** — MockProvider is hash-deterministic on the prompt, so the only way to vary the per-trial output is to perturb the prompt. We append `[trial_seed=N]` to the query. Both FinRoot and the baselines see the same perturbation for trial N, so the comparison stays apples-to-apples. Reproducibility invariant: same `(task, trial, base_seed)` always → same output.
- **2026-06-20 · wave-6/01-frb-expansion** — Adding 4 new domains (insurance, estate_planning, behavioral, international) to `data/gold/frb_questions.json` required updating `tests/unit/test_frb_bank.py` VALID_DOMAINS set. Without this, the `test_valid_domain` test would reject any question with a new domain. The contract's domain list in § FRB task shape was not exhaustive — it listed "portfolio | risk | tax | news_impact | cashflow | credit | general" as examples but the test enforced a closed set. Future expansion must update VALID_DOMAINS when adding new domains.
